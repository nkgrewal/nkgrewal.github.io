But how can we API harder?

We found that developers were experiencing a good deal of friction signing up for our services. The API products themselves were developing with the business, as well, so we needed a better way to update them throughout the system.

Origin story

An example of the original sign-up flow we were working with was as follows:

  1. Request API key via a landing page / API section of Pro site, depending on channel of arrival.
  2. Create an account to ensure our data isn't abused.
  3. Verify email for account access.
  4. Log in to the Developer Dashboard to test and run queries.

How do we even

We narrowed our focus to areas with the most payoff :

  • Simplify initial info we request of users.
  • Make the Developer Dashboard more approachable for new users.
  • Create a template to communicate API products more effectively.

Creeping on users

All of us had our assumptions about leaky points in the sign-up flow. The best way to vet them out was to check in with users. We utilized Full Story to watch where users would drop off during sign-up and talked to a handful of customers that went through the process. We also ran the current sign-up flow past a few of our internal developers, most which hadn't used our API service until then, to confirm suspected pain points.

The conclusion was that there was quite bit of friction over the information required to start, the length of the first form, verifying email, and not really knowing where to start testing out the API once they hit the dashboard. Product put together a Google doc going over goals, areas of focus, cost, benefit, and etcetera pros/cons for the team to collaborate over.

Charting the wild seas of registration

Whitepages instated SSO for all its products earlier in the year to centralize user accounts across consumer and B2B. It added a level of complexity for this project, as we only had so much control over what we asked of users. Certain info requirements for account creation were inflexible or would cost too much to change.

For Pro itself, we already had a multitude of people signing up under different paths, including free API trial, paid API service, Pro Web trial, paid Pro Web service, and a Pro Web viral sign-up campaign. Each path shared a lot of the same elements but with different treatments. To help the product and engineering team step back and conceptualize the entire registration UX, we created a set of diagrams per path (5 total) for the teams.

I started with the ideal flow for each user based on our research. The product and engineering teams brought in feedback and ideas to better modulate it. We settled on 2 iterations for each path to deal with the amount of info buy-in we asked from users. One version had all the form elements in one step, and the other version broke it into a step-by-step progression. We would test the two against each other.

I'm too lazy to build all this in Illustrator

To flesh out the building blocks we would need as quickly as possible, I sketched out a prototype of pages, which product uploaded and wired up in the InVision prototyping app for us to navigate. This helped the product and engineering teams get a better sense of what the real flow would feel like to users clicking through, including elements and messaging, while having the ability to toy with the flow and leave comments. It also made it easier to garner critical feedback before investing too much time in higher fidelity mockups.
